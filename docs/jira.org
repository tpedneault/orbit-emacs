#+title: jira sync

* Scope

This first Jira pass is read-only:
- fetch issues from Jira Server/Data Center
- sync them into =jira.org=
- open the browser issue from Org

Write actions now supported:
- explicit comments
- explicit transitions
- explicit worklog submission from Org clocks

Still not automatic:
- Org TODO changes do not push to Jira
- sync still preserves local =Notes=

* Setup

Set these in your user-local =config.el=:

- =orbit-user-jira-base-url=
- =orbit-user-jira-api-prefix=
- =orbit-user-jira-project-key=
- =orbit-user-jira-username=
- =orbit-user-jira-jql=
- =orbit-user-jira-org-file=
- =orbit-user-jira-token-command= or =orbit-user-jira-pat-env=

Defaults:
- API prefix: =/rest/api/2=
- Org file: =jira.org= under your Org directory

Authentication:
- keep the PAT out of the repo
- either fetch it with =orbit-user-jira-token-command=
- or point =orbit-user-jira-pat-env= at an environment variable name

Example:

#+begin_src emacs-lisp
(setq orbit-user-jira-base-url "https://jira.example.com")
(setq orbit-user-jira-project-key "PROJ")
(setq orbit-user-jira-username "thomas")
(setq orbit-user-jira-pat-env "JIRA_PAT")
#+end_src

* JQL

If =orbit-user-jira-jql= is set, orbit-emacs uses it directly.

Otherwise it builds:

#+begin_example
project = <project-key> AND assignee = <username> AND resolution = Unresolved ORDER BY updated DESC
#+end_example

That means v1 sync is intentionally scoped to:
- one configured project
- issues assigned to the configured user

* Commands

| Key         | Action                | Notes                     |
|-------------+-----------------------+---------------------------|
| =SPC n j c= | =mod-jira-add-comment= | add comment after confirmation |
| =SPC n j i= | =mod-jira-import-issue= | import one issue by key or number |
| =SPC n j s= | =mod-jira-sync=       | fetch and refresh =jira.org= |
| =SPC n j r= | =mod-jira-refresh-issue= | refresh current issue subtree only |
| =SPC n j t= | =mod-jira-transition-issue= | choose and apply a transition |
| =SPC n j o= | =mod-jira-open-issue= | open issue at point in browser |
| =SPC n j w= | =mod-jira-log-work=   | submit one Org CLOCK as a Jira worklog |
| =SPC n j W= | =mod-jira-log-work-manual= | submit a manual Jira worklog |
| =SPC n j J= | =mod-jira-open-file= | open =jira.org= in the notes workflow |
| =SPC n j j= | =mod-org-capture-journal= | journal capture stays here |

Import input:
- full key like =PROJ-75=
- bare number like =75=

If you enter a bare number, orbit-emacs prefixes it with =orbit-user-jira-project-key=.

All write actions are explicit and confirmed before the request is sent.

* Org Sync Shape

Each issue becomes one heading in =jira.org=.

Synced headings are agenda-friendly:
- they get a =:JIRA:= tag
- they get an Org TODO keyword mapped from the Jira status
- they keep the original Jira status in =JIRA_STATUS=

Status mapping:
- To Do / Open / Reopened -> =TODO=
- In Progress -> =IN-PROGRESS=
- Blocked / Waiting -> =WAIT=
- Done / Closed / Resolved -> =DONE=
- unknown status -> =TODO=

Heading shape:

#+begin_example
* TODO ABC-123 Summary :JIRA:
#+end_example

Managed content under each issue:

#+begin_example
** Description
** Comments
** Notes
#+end_example

Managed by sync:
- =Description=
- =Comments=

User-editable and preserved across refresh:
- =Notes=

That means you should keep local notes under =** Notes=.
Sync may replace =Description= and =Comments= on every refresh, but it will not overwrite =Notes=.
The single-issue refresh command updates only the issue subtree at point and leaves other Jira issue subtrees unchanged.

Stored properties include:
- =JIRA_KEY=
- =JIRA_STATUS=
- =JIRA_ASSIGNEE=
- =JIRA_PRIORITY=
- =JIRA_UPDATED=
- =JIRA_URL=

Refreshing keeps local note text already written under =** Notes=.
Existing non-Jira tags on a synced heading are preserved, and =:JIRA:= is kept without duplication.

If you have older freeform content directly under the issue heading, orbit-emacs will carry it forward into =Notes= on refresh.

Because =jira.org= lives under your Org directory by default, synced Jira items appear in the agenda automatically.
When you clock a Jira issue, the active clock also appears in the modeline
using the Jira key only, for example = [ABC-123] 0:32 =.

Why Jira items were easy to miss before:
- on a fresh setup, =jira.org= might not exist yet
- if the file did not exist, Org agenda discovery had nothing to include

orbit-emacs now ensures =jira.org= exists under the Org directory early, so imported or synced Jira items are visible to the agenda/dashboard.

* Write Actions

Endpoints used:
- add comment:
  - =POST /issue/<KEY>/comment=
- transition issue:
  - =GET /issue/<KEY>/transitions=
  - =POST /issue/<KEY>/transitions=
- worklog:
  - =POST /issue/<KEY>/worklog=

Comments:
- =SPC n j c=
- prompts for comment text
- asks for confirmation
- refreshes the issue afterward

Transitions:
- =SPC n j t=
- fetches live transitions from Jira for the current issue
- prompts with transition names
- asks for confirmation
- refreshes the issue afterward

Worklogs:
- =SPC n j w=
- uses a closed Org =CLOCK= entry under the current Jira issue
- prompts for a worklog comment describing what was done during that logged time
- asks for confirmation
- submits only once
- refreshes the issue afterward

Clock navigation:
- =SPC n c g=
- jumps directly to the active Org clock
- Jira clocks keep the shortened modeline display with only the Jira key

Manual worklogs:
- =SPC n j W=
- does not require an Org =CLOCK= entry
- prompts for time spent like =15m=, =1h30m=, or =3h=
- prompts for a worklog comment describing what was done
- prompts for a start date/time
- asks for confirmation
- refreshes the issue afterward

Open Jira Org file:
- =SPC n j J=
- opens the configured =jira.org=
- prefers the notes context when available

Journal capture:
- =SPC n j j=
- stays on this binding intentionally
- =SPC n j J= is used for =jira.org= so the journal workflow is not broken

Duplicate worklog protection:
- after a successful submission, orbit-emacs adds a local marker line after the CLOCK entry
- the same CLOCK entry will not be offered again

Org clock workflow:
1. clock in on a Jira issue heading
2. clock out
3. run =SPC n j w=
4. enter a worklog comment describing what you did
5. pick the CLOCK entry if more than one is available
6. confirm the worklog submission

From agenda:
- =SPC m c i= clocks into the Jira item at point
- the modeline still shows the shortened Jira clock as = [ABC-123] 0:xx =
- =SPC m c g= jumps to the active clocked task

Manual worklog workflow:
1. put point on a Jira issue heading
2. run =SPC n j W=
3. enter the time spent
4. enter a short description of the work
5. confirm the Jira worklog submission

* Troubleshooting

- Missing config:
  - set the Jira base URL and token source
- Missing default JQL inputs:
  - set =orbit-user-jira-jql=
  - or set both =orbit-user-jira-project-key= and =orbit-user-jira-username=
- HTTP error on sync:
  - confirm the base URL, API prefix, PAT, and Jira Server/Data Center endpoint
